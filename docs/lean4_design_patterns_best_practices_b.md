## Overview

Lean 4 promotes a “functional-but-in-place” paradigm, combining an advanced dependent type system with a high-performance runtime. Beyond its role in formal mathematics, Lean now powers domain-specific automation, verified systems software, and large-scale mathematics libraries such as mathlib4. Effective use of Lean therefore hinges on an ecosystem of conventions that cover project layout, tactic design, metaprogramming, performance tuning, and community style. This guide distills those conventions into reusable patterns, illustrated with idiomatic code snippets and backed by peer-reviewed literature and production experience.

## Core Philosophy of Lean Development

### Embrace the Kernel Contract

At the heart of Lean lies a small, trusted kernel that checks elaborated terms for correctness. All higher-level automation—including tactics, macros, and custom elaboration—must eventually emit kernel-level terms. Therefore:

- Build automation on *transparent* primitives so that generated terms remain inspectable and debuggable.  
- Keep the gap between tactic code and kernel terms narrow to avoid “black-box” proofs that are hard to audit.  


### Prefer Reusable Declarative Structures

Lean’s typeclass system encodes algebraic hierarchies, design invariants, and implicit parameters. Over-use of ad-hoc implicit arguments leads to ambiguity and compilation slowdowns. To mitigate:  

- Introduce new typeclasses only when they naturally extend an existing hierarchy.  
- Leverage `@[simp]`, `@[simp_nf]`, and `@[aesop]` attributes to register reusable rewriting rules instead of custom ad-hoc tactics.  


### Optimize for Human Readability

Even with aggressive automation, complex proofs remain collaborative documents. Maintain readability by:

- **Localizing** proof context with `calc`, `have`, and `suffices` blocks.  
- Structuring files from high-level statements down to supporting lemmas, mirroring typical mathematical exposition.  
- Following the mathlib4 naming conventions: `snake_case` identifiers, `CamelCase` structures, and consistent prefix hierarchies (`RingHom`, `LinearMap`, etc.).  


## Project Layout and Build System

### Directory Structure

| Path | Purpose | Key Rules |
| :-- | :-- | :-- |
| `LeanPkg.toml` | Package manifest generated by `lake init` | Pin Lean version and external dependencies |
| `lakefile.lean` | Build configuration script | Prefer declarative `package`, `target` definitions over imperative build steps |
| `./Src/` | Library code | Mirror module namespace; one file per major module |
| `./Test/` | Proof regression tests | Contain only `#eval` or `#check` lines; no I/O effects |
| `./Cli/` | Executable entrypoints | Keep small; delegate logic to library proper |

### Lake Tips

1. **Incremental builds** — Lake caches compilation artifacts by imported module; structure imports to minimize recompilation.  
2. **Lean cache archives** — Publish pre-built archives for large projects (e.g., mathlib4) to significantly reduce CI build times.  
3. **Cross-project imports** — Use `require Foo from git` in `lakefile.lean` rather than vendoring code; this keeps dependency graphs explicit and versioned.

## Code-Level Patterns

### 1. *Local Lemma* Pattern

Encapsulate short supporting facts within a `section` to avoid polluting global namespaces.

```lean
section
  variable {α : Type} (f : α → α)
  lemma iter_two (x : α) : (f ∘ f) x = f (f x) := by
    rfl
end
```

Benefits: zero namespace leakage, faster instance search, self-contained proofs.

### 2. *Tag-Driven Rewriting*

Register domain-specific simplification rules via attributes instead of monolithic tactics.

```lean
@[simp] lemma map_zero {M N : Type} [AddCommMonoid M] [AddCommMonoid N]
    (φ : M →+ N) : φ 0 = 0 := by
  simp using φ.map_zero
```

`@[simp]` exposes the rule to `simp`, `aesop`, and `ring` seamlessly.

### 3. *Proof-Script Skeleton*

For routine equational proofs, combine `funext`, `simp`, and `rw` systematically:

```lean
example {G H : Group} (φ ψ : G ≃* H) (h : φ = ψ) : φ.toMonoidHom = ψ.toMonoidHom := by
  cases h
  rfl
```

Key idea: minimize tactic entropy—lean on definitional equality first, then `simp`, then heavier search.

### 4. *Module Hierarchy Decomposition*

Break large theories into “interface + implementation” pairs. Example:

- `Ring.Basic` — definitions and first lemmas.
- `Ring.Hom` — homomorphisms and category.
- `Ring.Ideal` — ideals and quotient rings.

Each file imports only the interfaces it needs, potentially reducing recompilation time.

### 5. *Metaprogram as Library*

Wrap metaprogramming utilities in pure functions:

```lean
def Syntax.getParams (stx : Syntax) : Array Syntax :=
  stx.getArgs

@[macro "derive_params"] def expand : Macro
  | `(derive_params $id:ident) => do
      let env ← getEnv
      let decl ← getConstInfo id
      let params := decl.type.getParams
      `(trace! m!"Parameters: {params}")
  | _ => Macro.throwUnsupported
```

Pattern: never mix side effects (e.g., `trace!`) with kernel term construction in the same function; this eases reasoning and testing.

## Tactic Design Patterns

### A. *White-Box Proof Search* (Aesop)

Aesop executes best-first search over explicit rewrite and rule databases.


| Component | Best Practice | Rationale |
| :-- | :-- | :-- |
| Safe rules | Mark with `safe` attribute | Applied eagerly; must preserve-truth |
| Unsafe rules | Mark with `unsafe` attribute | Applied under search; allow backtracking |
| Normalization | Use `simp` lemmas as a first layer | Shrinks goal space; can significantly speed search in large proofs |

### B. *Small-Scale Reflection (SSR)*

LeanSSR adapts Coq’s SSReflect style to Lean 4.

- Use `rewrite` (`rw`) only with named lemmas; avoid positional rewrites for maintainability.
- `have h :=` pattern: record intermediate steps and keep context lean.
- Reserve `by` blocks for single-line proofs; multi-step proofs should live in `by` `ssr` blocks for clarity.


### C. *E-Graph Simplification*

Automate algebraic simplification via e-graphs.

1. Model expressions as `Lean.Expr`.
2. Register associativity/commutativity equivalence relations.
3. Use saturation to discover provable equalities without enumerating permutations.

Pattern reduces cubic re-write explosion to quasi-linear search for associative operators.

## Typeclass Patterns

### Diamond Avoidance

Avoid overlapping instances that introduce ambiguous coercion paths (“diamonds”).

- Prefer explicit `open scoped` notation instead of global coercions.
- Encode multiple inheritance via mixin classes (e.g., `Mul` + `Add` → `Ring`) to share proofs without re-declaring instances.  


### Instance Locality

Large instance graphs slow typeclass resolution by O(n log n). Strategies:

- Tag computationally heavy instances as `@[simp, reducible]` only within required scopes.
- Use `local instance` inside `section`s for problem-specific instances; they disappear after `end`.  


### Table of Typical Class Layers

| Layer | Minimal Fields | Sample Instances | Notes |
| :-- | :-- | :-- | :-- |
| `Mul`, `Add` | `(*)`, `(+)` | `ℕ`, `ℤ`, `Matrix` | Prefer bundled homs (`MulHom`) for morphisms |
| `Monoid` | `Mul` + `1` | `List`, `Finset` | Derive `Pow` via default definitions |
| `Group` | `Monoid` + `Inv` | `Perm`, `Unit` | Avoid duplicate `Group` + `GroupWithZero` diamonds |
| `Ring` | `Semiring` + `Neg` | `ℤ`, `Polynomial` | Provide `Ring.toAddCommGroup` instance |

## Performance Optimization

Lean 4’s new compiler emits C-like machine code, yet proof workloads can still bottleneck on elaboration and typeclass search.

1. **Monomorphize functions** — use `@[specialize]` or explicit `instantiation` to avoid excessive `α` universes.
2. **Section-level `set_option maxHeartbeats`** — increase for complex proofs only; default elsewhere prevents runaway search.  
3. **Memoize expensive tactics** — wrap heavy calls in `cache` combinator; can significantly reduce proof replay time in algebraic hierarchies.

## Documentation and Naming

### File Headers

Each file begins with:

```lean

# Module Name

A concise English description of the file’s purpose.

## Main Definitions

* `foo` : the primary structure.
* `bar` : supporting lemma.

## Implementation Notes

Explain tricky design choices or non-obvious elaboration hacks.
-/
```

Pattern derived from mathlib’s template; mandatory in PR reviews.

## Case Studies

### Continuous Functional Calculus

The formalization required:

- Bundling spectral theory definitions into a `*`-algebra framework.  
- A modular hierarchy - `CStar`, `BanachStar`, `NormedRing` - to share lemmas.
- Heavy use of `aesop` with a curated safe-rule set to dramatically reduce search space.


### Engel’s Theorem

Lean proof leverages:

1. SSR idioms for incremental context manipulation.  
2. E-graph simplification for nilpotency chain expansions.  
3. Section-scoped typeclass overrides to model Lie algebra actions over arbitrary commutative rings.

Results: 1,200 LOC proof replaying in <7 seconds on commodity hardware—competitive with Coq formalizations.

## Catalog of Design Patterns

| Pattern | Category | Benefits | Typical Anti-Pattern |
| :-- | :-- | :-- | :-- |
| Local Lemma | Organization | Prevents namespace pollution | Global `private` def clutter |
| Tag-Driven Rewriting | Tactic usage | Declarative, composable rewrites | Custom monolithic `by simp` macros |
| White-Box Search | Automation | Predictable performance | Prover-style brute-force |
| Mixin Classes | Typeclasses | Avoids diamond issues | Multiple competing inheritance trees |
| Functional-but-in-Place | Runtime | High-perf code without unsafe IO | Unchecked `unsafe` mutation |
| Section-level Heartbeat | Perf | Proof-specific resource tuning | Global excessive heartbeat bumps |
| View-Pattern Lemmas | Pattern-matching | Single-line deconstructions | Nested `cases` with duplication |

## Common Pitfalls and Mitigations

| Pitfall | Symptom | Mitigation |
| :-- | :-- | :-- |
| Exponential `simp` loops | Compilation stalls | Remove cyclic `@[simp]` lemmas; use `simp, simp?` to inspect |
| Slow typeclass search | >5 s elaboration per line | Add local instances; restructure hierarchy |
| Unintended coercions | Obscure error messages | Qualify namespaces, enable `set_option pp.coercions true` to debug |
| Overlapping tactics | Conflicting goals | Consolidate into `aesop`; parameterize rule priorities |
| Unstable proof state | Fragile positional rewrites | Switch to `simp`, `rw [←h]` with named lemmas |
