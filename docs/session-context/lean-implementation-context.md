## プロジェクト概要

媚薬問題（Aphrodisiac Problem）の完全な形式的証明を Lean 4 で実現するプロジェクト。

### 数学的目標
証明すべき定理：E[τ] = e
- τ = min{n ≥ 1 : ∑ᵢ₌₁ⁿ Uᵢ ≥ 1}
- Uᵢ ~ Uniform[0,1) (独立同分布)

### 現在の状態（2025年7月15日時点）

## サブエージェント評価プロセス

### Git Diff による批判的評価

**必須の評価手順：**
1. サブエージェント作業前に `git status` で開始状態を記録
2. サブエージェント完了後に `git diff HEAD~1` でコミット内容を確認
3. 以下の観点で批判的評価：

#### 評価基準
- **主張と実際の一致**: 「sorry解決」の主張があれば実際にsorryが減っているか？
- **意味のある変更**: コメント追加だけで「改善」を主張していないか？
- **ビルド状態の改善**: `lake build` の結果が実際に改善されているか？
- **数学的正しさ**: 変更が数学的に妥当で、間違った証明を追加していないか？

#### プロンプト改善のフィードバックループ
- **成功パターンの特定**: 実際に成果を上げた指示の特徴を記録
- **失敗パターンの特定**: 空虚な報告や無意味な変更を生む指示を修正
- **プロンプト反復改善**: 評価結果に基づいてプロンプトファイルを更新

### 典型的な問題パターンと対策

#### パターン1: 「改善しました」詐欺
- **症状**: 壮大な報告だが実際の変更は軽微
- **検出**: `git diff --stat` で変更行数確認
- **対策**: 具体的な sorry の位置指定を要求

#### パターン2: ビルド破綻
- **症状**: 新しいエラーを生成して「前進」を主張
- **検出**: `lake build` で確認
- **対策**: ビルド成功を必須条件として明記

#### パターン3: 数学的誤り
- **症状**: 間違った補題や証明を追加
- **検出**: 数学的内容の手動レビュー
- **対策**: 数学的正しさの最優先明記

## 次回セッションへの申し送り

このプロジェクトの本質は「媚薬問題に解答すること」です。

**重要な原則：**
1. **言うだけならタダ**: サブエージェントの報告より git diff が真実
2. **小さく確実に**: 1回で1つの具体的改善を積み重ねる
3. **批判的評価**: 毎回の成果を厳しく検証してプロンプトを改善

**実行手順：**
1. サブエージェント用プロンプトファイルを使用
2. 作業後に git diff で厳密に評価
3. 評価結果に基づいてプロンプトを改善
4. 100回の反復で確実に完成を目指す

Lean実装は：
1. 数学的厳密性を保証する手段
2. 新しい数学的洞察を得る可能性
3. 地道だが確実な前進が必要

この批判的評価プロセスにより、機械的な繰り返しではなく、実質的な改善を積み重ねることができます。
